# Gua 待修问题

以下问题已知且决定暂不处理或永不处理。审查代码时无需重复提出。

---

1. **`SetProcessDPIAware` 过时**
   windows crate 0.62 中没有导出 `SetProcessDPIAwareContext` 等新 API，旧 API 在 Win10/11 上正常工作。
   **不处理**：旧 API 在目标系统上功能正常，无实际影响。

2. **`RegisterHotKey` / `SetFocus` 重复 extern 声明**
   windows crate 中对应符号名不明确，无法直接替换。
   **不处理**：目前不影响编译和运行。

3. **滚动条**
   不加。滚动条实现难度高（该显示时消失、该固定时变短、逐条滚动和翻页的区别复杂），并且之前多次尝试实现失败。
   **永不处理**。

4. **`AddFontResourceExW` 返回值未检查**
   `state.rs` 调用后未检查返回值（0 表示失败），后续用读取到的 family name 创建字体可能因字体未注册而失败。
   **暂不处理**：触发概率低（需要字体文件损坏或注册失败），且依赖 format=1 字体才能有效验证，暂时无法测试。

5. **配置解析允许空 key / 空 value**
   `config.rs` 中 `= value`（key 空）或 `key =`（value 空）仍会生成 Entry，可能导致匹配异常。
   **暂不处理**：正常编辑配置不会产生空 key/value 行；即使出现也只是匹配不到或执行无果，不会崩溃。

6. **`entry_type` 中路径判断依赖 CWD**
   `state.rs` 用 `Path::new(val).is_dir()` 判断文件夹，如果 val 是相对路径，结果取决于进程工作目录，可能存在误判。
   **暂不处理**：所有条目都带 `[分类名]`，实际走不到 `entry_type`；且配置中用绝对路径时不受 CWD 影响。

7. **`#![windows_subsystem = "windows"]` 下 `panic.log` 可能写入失败**
   如果当前目录不可写（如 Program Files），panic 日志写不入文件。
   **暂不处理**：影响不大，panic 通常发生在开发阶段而非日常使用。未来可用 `%TEMP%/gua-panic.log` 做兜底。

8. **WM_CHAR 无法输入 emoji 等辅助平面字符**
   `wndproc.rs` 用 `char::from_u32` 处理 WM_CHAR，surrogate pair 会被过滤。
   **暂不处理**：搜索框不应输入 emoji，过滤掉是正确行为。

9. **`entry_type` 先判 .exe 再判 URL**
   如果一条记录同时满足两者条件，分类标成「程序」而不是「网址」。
   **不处理**：URL 不会以 `.exe` 结尾，实际不可能触发。

10. **`dispatch_wndproc` 遍历所有插件**
    窗口消息走到默认分支时，会挨个问每个插件要不要处理。如果未来装几十个插件，理论上每条消息要多跑几十次空判断。
    **暂不处理**：走到这个分支的消息都是低频消息（尺寸变化、DPI 变化等），打字、按键、绘图等高频消息在此之前已被拦截。每次判断开销不到 1 微秒，以目前插件数量完全无影响。如果将来插件超过 20 个再考虑优化。

11. **PLUGINS.r()/w() 别名 UB**
    `plugin.rs` 审计认为 `ST<T>::r()`（返回 `&T`）和 `w()`（返回 `&mut T`）可能同时存活导致 UB。
    **非 bug**：经逐行审查所有 `PLUGINS.r()`/`PLUGINS.w()` 调用点，每次返回的都是临时引用，在分号处或作用域结束时已 drop，不存在 r() 和 w() 引用同时存活的情况。`ST` 的 `r()`/`w()` 标记为 `unsafe` 允许调用方犯错，但当前所有调用点都正确遵守了不重叠的契约。

12. **`[profile.release]` 加 `panic = "abort"` 减体积**
    审计建议加 `panic = "abort"` 省 ~10-20% 体积。
    **不可行**：代码中 7 处使用 `std::panic::catch_unwind`（`settings.rs` 1 处保护 D2D/DWrite 异常 + `plugin.rs` 6 处隔离插件 panic）。加 `panic = "abort"` 后 `catch_unwind` 失效，任一插件 panic 会导致整个进程崩溃。若以后需要压缩此体积，需先重新设计插件错误处理（改用 `Result` 返回类型），属于独立大工程。
